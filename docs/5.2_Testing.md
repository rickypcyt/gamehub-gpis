## 5.2.1 Plan de Testing

### Alcance

El plan de pruebas cubre todas las funcionalidades implementadas en la plataforma GameHub, organizadas por módulos:

| Módulo | Funcionalidades a probar |
|--------|------------------------|
| **Autenticación y Autorización** | Registro de usuarios, login/logout, gestión de sesiones JWT, control de roles (admin, redactor, colaborador, suscriptor), protección de rutas |
| **Gestión de Noticias** | CRUD de noticias, filtrado por estado (published/featured), paginación, validación de slugs únicos |
| **Catálogo de Juegos** | Listado con filtros (género, plataforma), ordenación por puntuación, CRUD de juegos (admin/redactor) |
| **Blog y Comentarios** | CRUD de posts de blog, sistema de comentarios con verificación de autenticación, validación de contenido |
| **Perfil de Usuario** | Visualización y actualización de perfil propio, gestión de campos opcionales |
| **Contacto** | Envío de mensajes mediante formulario, validación de campos, almacenamiento en base de datos |
| **Panel de Administración** | Listado de usuarios (solo admin), gestión de contenido según rol, redirección post-login |
| **Internacionalización** | Cambio de idioma (ES/EN), renderizado de contenido traducido |
| **UI/UX Responsive** | Visualización correcta en dispositivos móviles, tablets y escritorio |

### Tipos de Prueba

| Tipo | Descripción | Aplicación en GameHub |
|------|-------------|----------------------|
| **Unitarias** | Prueban unidades individuales de código (funciones, componentes) aisladas | Validación con Zod (`registerSchema`, `newsSchema`, `contactSchema`), helpers de `auth-utils.ts`, queries de Neon |
| **Integración** | Prueban la interacción entre módulos del sistema | Comunicación Frontend ↔ API Routes ↔ Base de datos (Neon), flujo completo de autenticación (NextAuth + bcrypt + JWT), creación de noticia con revalidación de caché |
| **Sistema** | Prueban el sistema completo como un todo | Flujo end-to-end: usuario se registra → inicia sesión → crea comentario → visualiza perfil → cierra sesión |
| **Usuario (UAT)** | Pruebas de aceptación con escenarios reales de uso | Validación de redirección según rol tras login, verificación de protección de rutas, comprobación de responsive design |

### Criterios de Aceptación

Para que un caso de prueba se considere **superado**, debe cumplirse lo siguiente:

1. **Funcionalidad**: La salida real coincide con la salida esperada en todos los escenarios normales y de error documentados.
2. **Códigos HTTP**: Las respuestas de la API deben devolver los códigos de estado correctos (`200` OK, `201` Created, `400` Bad Request, `401` Unauthorized, `403` Forbidden, `404` Not Found, `409` Conflict, `500` Internal Server Error).
3. **Validación de datos**: Los esquemas Zod deben rechazar datos inválidos con mensajes de error descriptivos.
4. **Seguridad**: Las rutas protegidas deben rechazar accesos no autenticados o con roles insuficientes.
5. **Rendimiento**: Las consultas a la base de datos deben completarse en menos de 2 segundos bajo carga normal (seed data de 60+ registros).
6. **Persistencia**: Los datos escritos en la base de datos deben ser recuperables de forma consistente.

## 5.2.3 Informe de Pruebas de Integración

### Objetivo

Verificar que los módulos del sistema interactúan correctamente entre sí: Frontend (Next.js App Router) ↔ API Routes ↔ Base de datos (Neon PostgreSQL) ↔ Sistema de autenticación (NextAuth.js + bcrypt).

### Escenarios de Integración Probados

#### 1. Integración: Frontend Login → API Auth → Neon DB

**Flujo**: Usuario completa formulario en `/login` → `signIn("credentials", ...)` → API Route `[...nextauth]` → Query a Neon → bcrypt compare → JWT Session → Redirección según rol.

**Evidencia (logs del servidor de desarrollo)**:

```
[next-auth][debug][CREATE_SESSION] Session created for user: usr_abc123
[next-auth][debug][JWT] Token signed with role: "redactor"
POST /api/auth/callback/credentials 200 in 245ms
GET /api/auth/session 200 in 12ms
[Neon] Query: SELECT * FROM profiles WHERE email = $1 -- params: ["redactor@gamehub.com"]
[bcrypt] Password verification: true (cost: 10)
```

**Resultado**: ✅ **APROBADO**. El flujo completo de autenticación funciona sin errores. La sesión JWT contiene el rol correcto y el middleware puede leerlo.
